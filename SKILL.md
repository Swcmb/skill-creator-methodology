# Skill 创建方法论

本技能提供将任意教材、规范、文档转化为 TRAE AI Skill 的通用方法论与标准模板。AI 在创建新 Skill 时必须遵循本流程。

## 一、适用范围

当出现以下任一情形时，使用本方法论：
- 用户要求将某份规范/教材/文档转化为 Skill；
- 用户要求"创建技能"、"编写 skill"、"制作 skill 文件"；
- 用户要求为已有规范编写可被 AI 遵循的指令集；
- 用户询问"如何写一个 skill"、"skill 怎么创建"。

不适用情形：
- 修改已有 skill 的小细节（直接编辑即可）；
- 创建普通代码文件而非 skill。

## 二、整体流程（5 步法）

```
调研格式 → 提炼要点 → 编写 SKILL.md → 编写 README.md → 打包交付
```

### 第 1 步：调研 Skill 规范格式

| 调研要点 | 说明 |
|---|---|
| 文件格式 | `SKILL.md`（Markdown + YAML frontmatter） |
| 必填字段 | `name`、`description` |
| 可选字段 | `metadata.internal`、`allowed-tools` |
| 全局存储位置 | `~/.trae/skills/<skill-name>/SKILL.md` |
| 项目存储位置 | `.trae/skills/<skill-name>/SKILL.md` |

**操作建议**：使用 search 子代理扫描文件系统，定位现有 skill 实例与官方文档，避免凭空猜测。

### 第 2 步：分析源材料，提炼规范要点

提炼策略：
1. **语法要素 / 可枚举规则**（可表格化）→ 用表格或代码块呈现；
2. **书写规范 / 原则性要求**（描述性）→ 用分点列表呈现；
3. **示例**（具象化）→ 保留原典例题，必要时补充完整示例。

**关键原则**：
- 保留原文"应当/必须/避免"等措辞的语气强度，区分"强制规则"与"建议"；
- 保留原典的术语和命名约定（如 `ElemType`、`Status` 等）；
- 不擅自添加原文未提及的规则。

### 第 3 步：编写 SKILL.md（核心文件）

按下方"三、SKILL.md 标准模板"编写。

### 第 4 步：编写 README.md（说明文档）

按下方"四、README.md 标准模板"编写。

### 第 5 步：打包与交付

1. 用 `tar -czf <skill-name>.tar.gz <skill-name>/` 打包；
2. 同步提供仓库级 README，便于 GitHub 发布；
3. 将文件复制到工作目录 `/workspace/` 方便用户下载预览。

## 三、SKILL.md 标准模板

```markdown
---
name: <skill-name>
description: <来源说明>。当用户<触发场景1>、<触发场景2>，或提及"<关键词1>"、"<关键词2>"时，使用本技能。要求 AI <执行要求>。
---

# <技能标题>

<引言：一句话说明本技能的来源与作用>

## 一、适用范围
- <触发情形1>
- <触发情形2>
- <不适用情形>

## 二、<规范要素1：可表格化>
<表格或代码块>

## 三、<规范要素2：可表格化>
<表格或代码块>

...

## N、<书写规范：分点描述>
1. <规范1>
2. <规范2>

## N+1、完整示例
<符合全部规范的完整代码示例>

## N+2、自检清单
- [ ] <检查项1>
- [ ] <检查项2>
```

### SKILL.md 编写要点

| 要点 | 说明 |
|---|---|
| **name** | 小写 + 连字符，语义清晰，体现来源与用途 |
| **description** | 触发关键，必须包含"何时触发"+"做什么" |
| **章节编号** | 中文数字（一、二、三）贴合中文规范文档风格 |
| **语法要素** | 优先用表格，可读性高于纯文字 |
| **书写规范** | 用分点列表，保留"必须/不得/建议"的语气强度 |
| **示例** | 保留原典例题，补充完整可参照示例 |
| **自检清单** | 末尾加 checkbox，让 AI 输出前能自我校验 |

## 四、README.md 标准模板

```markdown
# <skill-name>

> <一句话定位>

## 📖 简介
<来源、作用>

## ✨ 功能特性
- <特性1>
- <特性2>

## 📁 仓库结构
\`\`\`
<skill-name>/
├── SKILL.md
└── README.md
\`\`\`

## 🚀 安装使用
### 全局安装（推荐）
<命令>

### 项目级安装
<命令>

### 手动安装
<说明>

## 🎯 触发场景
- <关键词/场景1>
- <关键词/场景2>

## 📚 参考来源
- <出处>

## 📄 许可证
MIT License
```

## 五、关键设计原则

| 原则 | 说明 |
|---|---|
| **触发条件要明确** | description 中写清"什么场景用"+"什么场景不用"，避免误触发 |
| **强制与建议分离** | "必须/不得" vs "建议/可以"，语气强度有区分 |
| **抽象与具象结合** | 表格给抽象规则，代码块给具象示例 |
| **保留原典风格** | 引用教材/规范原文时，保留其术语和命名约定 |
| **可自检** | 末尾清单让 AI 能在输出前校验自身是否符合规范 |
| **命名要语义化** | skill 名应体现来源和用途 |

## 六、自检清单

创建 Skill 完成后，请对照以下清单自检：

- [ ] frontmatter 是否包含 `name` 和 `description` 两个字段？
- [ ] `name` 是否小写 + 连字符，语义清晰？
- [ ] `description` 是否写明触发场景、关键词、执行要求？
- [ ] 是否有"适用范围"与"不适用情形"？
- [ ] 语法/规则要素是否优先用表格呈现？
- [ ] 书写规范是否用分点列表，保留"必须/建议"语气？
- [ ] 是否保留了原典的术语和命名约定？
- [ ] 是否提供了完整可参照的示例？
- [ ] 末尾是否有自检清单？
- [ ] 是否同步编写了 README.md？
- [ ] 文件是否存放于 `~/.trae/skills/<skill-name>/` 或 `.trae/skills/<skill-name>/`？
